## 1. 배경 및 목적

### 1.1 현재 한계 (As-Is)

v1 WMS는 **입고 주문·출고 주문을 각각 독립적으로 조회·상태 전이**하는 구조이다.

| 영역 | 현재 동작 | 문제 |
|------|-----------|------|
| 입고 | 시드 데이터로 주문이 미리 존재, 목록 조회·상태 전이만 가능 | **입고 등록(생성) 화면·API 없음** |
| 재고 | 재고 테이블·화면 없음 | 입고 확정이 실물 재고에 반영되지 않음 |
| 출고 | 시드 데이터로 주문이 미리 존재, 재고 검증 없음 | **가용 재고와 무관하게 출고 가능** |
| 연계 | 입고·출고 간 데이터 연결 없음 | 운영자가 “입고 → 재고 → 출고” 흐름을 추적·검증할 수 없음 |

### 1.2 목표 (To-Be)

센터·SKU 단위 **재고를 중심**으로 입고·출고 업무를 하나의 흐름으로 연결한다.

```
[입고 등록] → [검수·입고확정] → [재고 증가] → [재고 조회] → [출고 등록] → [피킹·출고확정] → [재고 감소]
```

운영자는 화면에서 다음을 수행할 수 있어야 한다.

1. 입고 주문을 **직접 등록**한다.
2. 입고 확정 시 **재고가 자동 반영**되는 것을 확인한다.
3. 재고 현황을 **센터·SKU별로 조회**한다.
4. 가용 재고 범위 내에서 **출고 주문을 등록**한다.
5. 출고 확정 시 **재고가 차감**되며, 부족 시 **명확한 오류**를 받는다.

### 1.3 범위

본 문서는 v1 기능(F1~F4) 위에 **신규 기능(F5~F7)을 추가**하는 추가 PRD이다.

- v1 PRD·REQ·UI·Task 문서는 **변경하지 않는다**.
- 신규 요구사항은 `REQ-F5`, `REQ-F6`, `REQ-F7` 등 **별도 문서**로 작성한다.
- v1 상태 전이 규칙(F4)과 기술 스택은 그대로 따르며, 입고·출고 확정 시 재고 연동 로직만 **코드·신규 REQ**에 반영한다.

## 4. 업무 흐름 (E2E)

### 4.1 정상 흐름

```mermaid
flowchart LR
    A[입고 등록 F5] --> B[검수시작 00→10]
    B --> C[검수완료 10→20]
    C --> D[입고확정 20→90 F1]
    D --> E[재고 증가 F6]
    E --> F[재고 조회 F6]
    F --> G[출고 등록 F7]
    G --> H[피킹시작 00→10]
    H --> I[피킹완료 10→20]
    I --> J[출고확정 20→90 F2]
    J --> K[재고 감소 F6]
```

### 4.2 재고 반영 시점

| 이벤트 | 재고 동작 | 비고 |
|--------|-----------|------|
| 입고 등록 (`00`) | 변동 없음 | 주문만 생성 |
| 입고 확정 (`20→90`) | **가용 재고 증가** | 라인별 `qty_confirmed` 반영 |
| 출고 등록 (`00`) | 변동 없음 (선택: 예약은 2차) | 등록 시 가용 재고 **검증만** |
| 출고 확정 (`20→90`) | **가용 재고 감소** | 라인별 `qty_picked` 반영 |

> 1차 구현: **출고 등록 시점**에 가용 재고 부족이면 등록 거부. **출고 확정 시점**에 재고 차감(동시성 재검증).

### 4.3 사용자 시나리오 (Happy Path)

1. 운영자가 **입고 등록** 화면에서 센터 `CTR01`, SKU `SKU-AAA` 100개 주문을 생성한다.
2. 입고 상세에서 **검수시작 → 검수완료 → 입고확정**을 진행한다.
3. **재고 현황** 화면에서 `CTR01` / `SKU-AAA` 가용 수량이 100 증가했음을 확인한다.
4. **출고 등록** 화면에서 동일 센터·SKU 30개 출고 주문을 생성한다.
5. 출고 상세에서 **피킹시작 → 피킹완료 → 출고확정**을 진행한다.
6. 재고 현황에서 가용 수량이 70으로 감소했음을 확인한다.
7. 대시보드·이력에서 입고·출고 주문 추적이 가능하다 (기존 F3 활용).

## 5. 기능 요구사항

### F5. 입고 등록

입고 주문을 **화면에서 신규 생성**한다. (v1 F1은 조회·전이만 제공)

| ID | 요구 | 상세 |
|----|------|------|
| F5-1 | 주문 생성 | 센터, 품목 라인(SKU, 예정 수량)을 입력하여 입고 주문을 등록한다. |
| F5-2 | 초기 상태 | 생성 시 `status_cd = 00`(주문생성)으로 저장한다. |
| F5-3 | 주문번호 채번 | 서버가 `IB-{센터코드}-{일련}` 또는 `IB-{YYYYMMDD}-{일련}` 형식으로 자동 발급한다. |
| F5-4 | 유효성 검증 | 센터·SKU 마스터 존재 여부, 수량 > 0, 라인 1건 이상 필수. |
| F5-5 | 생성 후 흐름 | 생성 완료 시 입고 상세 화면으로 이동하거나, 목록에 즉시 표시한다. |
| F5-6 | 검수 수량 | 검수완료·입고확정 시 라인 `qty_confirmed`를 설정한다. 1차: `qty_ordered`와 동일 자동 반영 허용. |

**v1 F1 변경:** 입고 목록 화면에 **[입고 등록]** 버튼 추가.

---

### F6. 재고 관리

센터·SKU 단위 **가용 재고**를 관리·조회한다.

| ID | 요구 | 상세 |
|----|------|------|
| F6-1 | 재고 마스터 | `(center_cd, sku_cd)` 단위로 `qty_on_hand`(가용 수량)를 관리한다. |
| F6-2 | 입고 반영 | 입고 확정(`20→90`) 트랜잭션 내에서 해당 라인 수량만큼 재고를 **증가**시킨다. |
| F6-3 | 출고 반영 | 출고 확정(`20→90`) 트랜잭션 내에서 해당 라인 수량만큼 재고를 **감소**시킨다. |
| F6-4 | 재고 이력 | 증감 시 `twms_inventory_hist`에 유형·참조 주문·수량·잔량을 기록한다. |
| F6-5 | 재고 조회 | 센터·SKU 필터로 재고 목록을 조회한다. 상품명·센터명 조인 포함. |
| F6-6 | 재고 상세 | 특정 센터·SKU의 현재 수량 및 최근 증감 이력을 조회한다. |
| F6-7 | 음수 방지 | 출고 확정 시 재고 부족이면 트랜잭션 롤백, HTTP 400과 명확한 메시지 반환. |
| F6-8 | 멱등·일관성 | 동일 입고 주문의 중복 확정으로 재고가 이중 반영되지 않도록 한다. (상태 전이 규칙 + 비즈니스 가드) |

**재고 이력 유형 (`inv_txn_type`):**

| 코드 | 명칭 | 발생 시점 |
|------|------|-----------|
| `INBOUND` | 입고 반영 | 입고 확정 |
| `OUTBOUND` | 출고 반영 | 출고 확정 |

---

### F7. 재고 연동 출고

출고 주문을 **가용 재고 기준으로 등록·처리**한다.

| ID | 요구 | 상세 |
|----|------|------|
| F7-1 | 출고 등록 | 센터, 품목 라인(SKU, 출고 수량)을 입력하여 출고 주문을 등록한다. |
| F7-2 | 재고 검증 (등록) | 등록 시점에 센터·SKU별 `qty_on_hand >= 출고 수량`을 검증한다. 부족 시 등록 거부. |
| F7-3 | 초기 상태 | 생성 시 `status_cd = 00`(주문생성). |
| F7-4 | 주문번호 채번 | 서버가 `OB-{센터코드}-{일련}` 또는 `OB-{YYYYMMDD}-{일련}` 형식으로 자동 발급. |
| F7-5 | 피킹 수량 | 피킹완료·출고확정 시 라인 `qty_picked` 설정. 1차: `qty_ordered`와 동일 자동 반영 허용. |
| F7-6 | 재고 검증 (확정) | 출고 확정 시 재고 재검증 후 차감 (동시 출고 대비). |
| F7-7 | 가용 재고 표시 | 출고 등록 화면에서 SKU 선택 시 **현재 가용 재고**를 표시한다. |
| F7-8 | 입고 연계 표시 | 재고 상세·출고 등록에서 해당 SKU의 최근 입고 확정 주문을 참조 링크로 표시 (선택, 1차 권장). |

**v1 F2 변경:** 출고 목록 화면에 **[출고 등록]** 버튼 추가. 기존 시드 주문은 유지하되, 신규 주문은 재고 검증 적용.

### 6.2 기존 테이블 활용 (변경 없음·확장 없음)

- `twms_inbound_order` / `_line` — 입고 등록·확정 수량
- `twms_outbound_order` / `_line` — 출고 등록·피킹 수량
- `twms_order_status_hist` — 상태 이력 (F4)

### 6.3 초기 데이터 정책

| 항목 | 정책 |
|------|------|
| 기존 시드 입고·출고 | 유지. 단, **기존 확정(`90`) 주문에 대해 재고 역산 INSERT는 1차 생략** (신규 확정분부터 재고 반영). |
| 재고 초기값 | `twms_inventory`는 빈 상태로 시작. 교육 시나리오는 F5 입고 등록부터 수행. |
| 선택 과제 | `06_init_inventory_from_seed.sql` — 기존 확정 주문 기준 재고 역산 (2차). |

```

```

## 9. 오류·예외 시나리오

| 시나리오 | 기대 동작 |
|----------|-----------|
| 출고 등록 수량 > 가용 재고 | HTTP 400, `"가용 재고가 부족합니다: {sku_cd} (가용 {n}, 요청 {m})"` |
| 출고 확정 시 재고 부족 (동시 처리) | HTTP 400, 트랜잭션 롤백, 상태 변경 없음 |
| 입고 확정 중 재고 반영 실패 | HTTP 500 또는 400, 입고 상태도 롤백 (단일 트랜잭션) |
| 존재하지 않는 SKU로 등록 | HTTP 400 |
| 라인 없이 주문 등록 | HTTP 400 |
| 수량 0 이하 | HTTP 400 |

## 10. 비기능 요구사항

| 구분 | 요구 |
|------|------|
| 트랜잭션 | 상태 전이 + 재고 증감 + 재고 이력 INSERT는 **단일 트랜잭션** |
| 동시성 | 재고 UPDATE 시 `(center_cd, sku_cd)` 행 기준 잠금 또는 `qty_on_hand` 조건부 UPDATE (`WHERE qty_on_hand >= ?`) |
| API 규격 | v1과 동일 — `Map<String, Object>`, snake_case, 공통 응답 래퍼 |
| SQL | MyBatis XML only |
| 가시성 | 재고 이력으로 입고·출고 주문과 추적 가능 |
| 교육용 | E2E 시나리오 1회(입고 등록 → 확정 → 출고 등록 → 확정)로 데모 가능해야 함 |
